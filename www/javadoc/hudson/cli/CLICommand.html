Base class for Hudson CLI.

How does a CLI command work

The users starts the "CLI agent" on a remote system, by specifying arguments, like "java -jar hudson-cli.jar command arg1 arg2 arg3". The CLI agent creates a remoting channel with the server, and it sends the entire arguments to the server, along with the remoted stdin/out/err.

The Hudson master then picks the right CLICommand to execute, clone it, and calls main(List, InputStream, PrintStream, PrintStream) method.

Note for CLI command implementor

  • Put Extension on your implementation to have it discovered by Hudson.
  • Use args4j annotation on your implementation to define options and arguments (however, if you don't like that, you could override the main(List, InputStream, PrintStream, PrintStream) method directly.
  • stdin, stdout, stderr are remoted, so proper buffering is necessary for good user experience.
  • Send Callable to a CLI agent by using channel to get local interaction, such as uploading a file, asking for a password, etc.

getName

public java.lang.String getName()
Gets the command name.

For example, if the CLI is invoked as java -jar cli.jar foo arg1 arg2 arg4, on the server side CLICommand that returns "foo" from getName() will be invoked.

By default, this method creates "foo-bar-zot" from "FooBarZotCommand".

run

protected abstract int run()
                    throws java.lang.Exception
Executes the command, and return the exit code.

Returns:
0 to indicate a success, otherwise an error code.
Throws:
AbortException - If the processing should be aborted. Hudson will report the error message without stack trace, and then exits this command.
java.lang.Exception - All the other exceptions cause the stack trace to be dumped, and then the command exits with an error code.
